# SecureZip for Mac - 要件定義書

## 1. プロジェクト概要

### 1.1 プロジェクト名
SecureZip for Mac

### 1.2 プロジェクトコード
SZ-2026

### 1.3 目的・背景

**目的**: コマンドラインの知識がなくても、ビジネスレベルのセキュアなファイル送付ができるMacアプリケーションを開発する。

**背景**: 現在、Mac環境でファイルを暗号化圧縮してメール送付するには、ターミナルでのコマンド操作や複数のアプリケーションを横断する必要がある。非エンジニアのビジネスユーザーにとってこれは大きな障壁であり、結果としてセキュリティが不十分なまま機密ファイルがやりとりされるケースが多い。

**解決したい課題**:
- 暗号化圧縮にターミナル操作が必要な現状の解消
- パスワード管理の煩雑さの解消（自動生成・自動保存・自動削除）
- ファイル暗号化からメール送信までの分断されたワークフローの統合
- パスワードの安全な別送（ファイルとパスワードを同一メールで送るリスクの排除）

### 1.4 対象システム
- **システム名**: SecureZip for Mac
- **対象範囲**: ファイルの圧縮・解凍・暗号化・パスワード管理・Gmail連携送信
- **関連システム**: Gmail API（Google Cloud Platform）、macOS Keychain Services

### 1.5 対象プラットフォーム・環境
- **OS**: macOS 13 Ventura 以降
- **デバイス**: Mac（Apple Silicon および Intel）
- **ブラウザ**: Safari（OAuth認証画面表示のみ使用）
- **開発環境**: Swift / SwiftUI / Xcode 15+

## 2. 現状分析

### 2.1 現状の業務フロー

現在、Macユーザーが機密ファイルを暗号化して送付する場合の一般的なフロー:

1. ターミナルを開き、zipコマンドで暗号化圧縮（`zip -e archive.zip file`）
2. パスワードを手動で決定・入力
3. メールアプリ（Mail.app等）を開き、圧縮ファイルを添付して送信
4. 別のメールでパスワードを手動で送信
5. パスワードをメモ帳やスプレッドシートで自己管理

### 2.2 現状の課題
| 課題項目 | 詳細 | 影響度 | 優先度 |
|----------|------|--------|--------|
| ターミナル操作が必要 | 非エンジニアにとって暗号化圧縮のハードルが高い | 高 | 高 |
| パスワード管理が煩雑 | パスワードを安全に生成・保管・共有する仕組みがない | 高 | 高 |
| ワークフローが分断 | 圧縮→メール送信→パスワード送信が別々のアプリで行われる | 中 | 高 |
| パスワードの安全性 | 弱いパスワードが使い回される、同一メールでパスワードを送付するリスク | 高 | 高 |
| 履歴管理の欠如 | いつ・誰に・どのファイルを送ったかの記録が残らない | 中 | 中 |
| macOS標準機能の不足 | Finderの圧縮機能にはパスワード暗号化オプションがない | 高 | 高 |

### 2.3 改善要求

- GUI操作のみで暗号化圧縮からメール送信まで完結するアプリケーションの提供
- 暗号学的に安全なパスワードの自動生成
- パスワードとファイルの自動別送によるセキュリティ向上
- 送付履歴の自動記録と一定期間後の自動削除によるセキュリティと利便性の両立
- App Sandbox内のローカル完結処理によるプライバシー確保

## 3. 要求事項

### 3.1 業務要求

#### 3.1.1 業務目標
- 非エンジニアのMacユーザーが、3ステップ以内で暗号化ファイルを安全に送信できる状態を実現する
- パスワード別送をデフォルト動作とし、セキュリティインシデントのリスクを低減する
- 送付履歴を自動管理し、業務の追跡可能性を確保する

#### 3.1.2 業務範囲
- ファイル・フォルダの圧縮および解凍
- AES-256暗号化によるパスワード付き圧縮（ZIP形式）
- 安全なパスワードの自動生成
- Gmail連携によるアプリ内メール送信
- パスワードの自動別送
- 送付履歴の記録・検索・自動削除
- パスワード・トークンのKeychain保管

#### 3.1.3 業務フロー（To-Be）
```
新しい業務フロー（SecureZip使用時）

1. SecureZipアプリを起動
2. ファイル/フォルダをドラッグ&ドロップ
3. 圧縮形式を選択（ZIP / TAR.GZ / TAR.BZ2 / TAR.ZST）
4. AES-256暗号化をON → パスワード自動生成（または手動入力）
5. 「送信」を選択
6. 送付先メールアドレスを入力（オートコンプリート対応）
7. 送信ボタン押下 → カウントダウン表示（キャンセル可能）
8. 本体メール自動送信 → パスワード通知メール自動別送
9. 送付履歴が自動保存 → 設定日数後に自動削除
```

#### 3.2.2 詳細機能仕様

##### F001: 圧縮
- **目的**: ファイル・フォルダを指定形式で圧縮する
- **概要**: ユーザーが選択したファイルまたはフォルダを、指定した圧縮形式（ZIP / TAR.GZ / TAR.BZ2 / TAR.ZST）でアーカイブファイルに圧縮する
- **入力**: 圧縮対象ファイル/フォルダ（1つ以上）、圧縮形式、出力先パス
- **処理**: libarchiveを使用したストリーミング圧縮処理。進捗バーをUIに表示
- **出力**: 指定形式の圧縮ファイル
- **例外処理**: ディスク容量不足時はエラーアラート表示。ファイルアクセス権限不足時は再選択を促す
- **前提条件**: App Sandboxでユーザーが選択したファイルへの読み取り権限が付与されていること
- **事後条件**: 圧縮ファイルが指定パスに作成される。圧縮ログがConsole.appで確認可能

##### F003: AES-256暗号化圧縮
- **目的**: 機密ファイルをパスワード付きで安全に圧縮する
- **概要**: ZIP形式でのAES-256暗号化圧縮。libarchiveの`zip:encryption=aes256`オプションを使用
- **入力**: 圧縮対象ファイル/フォルダ、パスワード（手動入力または自動生成）
- **処理**: パスワード強度チェック → libarchiveによるAES-256暗号化ZIP圧縮
- **出力**: AES-256暗号化されたZIPファイル
- **例外処理**: パスワードが強度要件を満たさない場合は警告表示（続行可能）。ZIP以外の形式で暗号化が要求された場合はエラー
- **前提条件**: 圧縮形式がZIPであること
- **事後条件**: 暗号化ZIPファイルが作成され、正しいパスワードでのみ解凍可能

##### F008: Gmail連携送信
- **目的**: アプリ内からGmail経由でファイルを送信する
- **概要**: OAuth 2.0認証を経てGmail APIを使用し、圧縮ファイルを添付したメールをバックグラウンドで送信する
- **入力**: 送付先メールアドレス、件名、本文、添付ファイル
- **処理**: MIMEメッセージ構築 → Base64URLエンコード → Gmail API `/messages/send` 呼び出し
- **出力**: 送信完了通知（成功 / 失敗）
- **例外処理**: 認証期限切れ時はRefresh Tokenで自動再認証。ネットワークエラー時はリトライボタン表示。25MB超過時はエラーアラートと分割圧縮の推奨
- **前提条件**: Gmail OAuth認証が完了していること。ネットワーク接続があること
- **事後条件**: メールが送信され、送付履歴がローカルに記録される

##### F009: パスワード別送
- **目的**: ファイル本体とパスワードを別メールで送付し、セキュリティを向上する
- **概要**: 暗号化ファイル添付メール送信後、数秒の間隔をあけてパスワード通知メールを自動送信する
- **入力**: 送付先メールアドレス、パスワード
- **処理**: 本体メール送信完了 → 設定秒数待機 → パスワード通知メール送信
- **出力**: 2通のメール（本体メール + パスワード通知メール）の送信完了通知
- **例外処理**: 1通目成功・2通目失敗の場合、パスワード通知メールのリトライを促す。パスワードは履歴から再取得可能
- **前提条件**: Gmail連携が完了していること。暗号化圧縮ファイルが存在すること
- **事後条件**: 2通のメールが送信される。送付履歴にパスワード送信状態が記録される

##### F010: 送信キャンセル
- **目的**: 誤送信を防止する
- **概要**: 送信ボタン押下後、ユーザー設定の秒数間カウントダウンを表示し、その間にキャンセル可能とする
- **入力**: キャンセルボタン押下（キャンセル時）
- **処理**: Swift Concurrency の Task を使用。カウントダウン中にキャンセルが押された場合は `Task.cancel()` で送信処理を即時中断
- **出力**: キャンセル成功通知 / カウントダウン終了後に送信実行
- **例外処理**: カウントダウン中にアプリが終了した場合は送信を実行しない（安全側に倒す）
- **前提条件**: 送信処理がカウントダウン段階であること（Gmail API呼び出し前）
- **事後条件**: キャンセル時は送信が行われず、ステータスが`cancelled`に更新される

### 3.4 品質要求

#### 3.4.1 信頼性
- **可用性**: スタンドアロンアプリのため、ネットワーク不通時もローカル機能（圧縮・解凍・パスワード生成）は100%利用可能
- **障害対応**: 圧縮処理中のクラッシュ時、不完全な出力ファイルを自動削除
- **データ整合性**: 履歴DBとKeychainの整合性を起動時にチェック。不整合検出時は自動修復または該当レコード削除

#### 3.4.2 使用性（ユーザビリティ）
- **操作性**: 非エンジニアユーザーが初回利用時にマニュアルなしで基本操作（圧縮・解凍）を完了できること
- **学習性**: Gmail連携を含む全機能を15分以内に習得可能であること
- **アクセシビリティ**: macOS VoiceOverに対応。キーボードショートカットでの主要操作対応

#### 3.4.3 効率性
- **処理効率**: ストリーミング処理により、圧縮対象ファイルサイズに関わらずメモリ使用量を一定に保つ
- **ストレージ効率**: 送付履歴のデータサイズは1レコードあたり1KB未満。パスワードはKeychainに保存しDBのサイズを抑制

#### 3.4.4 保守性
- **変更容易性**: MVVM + Service層によるレイヤード設計で、機能追加・変更時の影響範囲を局所化
- **診断性**: Unified Logging（os_log）による構造化ログ出力。Console.appでの診断が可能
- **試験性**: Service層のプロトコル化によりモック差し替えが容易。単体テストカバレッジ80%以上を目標

#### 3.4.5 移植性
- **環境適応性**: macOS 13 Ventura以降のすべてのバージョンで動作。Apple Silicon / Intel 両対応（Universal Binary）
- **インストール性**: App Storeからのワンクリックインストール。追加設定不要で基本機能が利用可能

### 3.5 セキュリティ要求

#### 3.5.1 認証・認可
- **ユーザー認証**: Gmail送信機能のみOAuth 2.0認証が必要。ローカル機能は認証不要
- **アクセス制御**: App Sandboxによりアプリのファイルアクセスをユーザー選択ファイルに限定
- **パスワードポリシー**: 自動生成パスワードは16文字以上、大文字・小文字・数字・記号を含む。手動入力パスワードは最低8文字以上を推奨（警告表示のみ、強制ではない）

#### 3.5.2 データ保護
- **暗号化**: ファイル暗号化はAES-256（libarchive）。パスワード・トークンの保管はKeychain Services（OS暗号化）
- **バックアップ**: ローカル完結のためサーバーサイドバックアップは不要。Time Machineによるユーザー側バックアップに委ねる
- **ログ管理**: 操作ログはUnified Loggingに出力。パスワード・トークン等の機密情報はログに出力しない

#### 3.5.3 プライバシー保護
- **個人情報保護**: 送付先メールアドレスはローカルにのみ保存。外部送信しない
- **データ収集**: アプリはアナリティクス・テレメトリデータを収集しない
- **OAuth権限**: Gmail APIスコープは`gmail.send`のみに限定し、メール読み取り・連絡先アクセス等は行わない

|------------|----------|--------------|---------------|
| 送付履歴 | 0 | 500件（想定：週10件送信） | 2,500件（自動削除で減少） |
| 送付先 | 0 | 50件 | 250件 |
| アプリ設定 | - | - | UserDefaultsで管理（件数概念なし） |

### 4.3 インターフェース仕様

#### 4.3.1 ユーザーインターフェース
- **画面構成**: NavigationSplitView（サイドバー + コンテンツ領域）のシングルウィンドウ構成
- **操作フロー**: サイドバーで機能選択 → コンテンツ領域で操作 → シートやオーバーレイで補助機能表示
- **入力規則**: メールアドレスはRFC 5322形式でバリデーション。パスワードは最低8文字。キャンセル秒数は1〜10秒の範囲

#### 4.3.2 外部システム連携
| 連携先システム | 連携方式 | データ形式 | 頻度 | 備考 |
|----------------|----------|------------|------|------|
| Gmail API | REST API (HTTPS) | JSON + Base64 MIME | ユーザー操作時 | スコープ: gmail.send のみ |
| Google OAuth Server | OAuth 2.0 + PKCE | JSON | 初回認証時・トークンリフレッシュ時 | カスタムURLスキームでコールバック |

#### 4.3.3 API仕様
- **認証方式**: OAuth 2.0 Bearer Token
- **データ形式**: JSON（リクエスト・レスポンス）、Base64URL（MIMEメッセージ）
- **エラーハンドリング**: HTTPステータスコードに基づく分岐処理。401時は自動トークンリフレッシュ。429時はリトライ。5xx時はユーザーに通知

### 4.4 運用要求

#### 4.4.1 運用時間
- **サービス時間**: スタンドアロンアプリのため24時間利用可能（ローカル機能）。Gmail送信はGmail APIの稼働に依存
- **メンテナンス時間**: App Storeアップデート配布時のみ（ユーザー任意）

#### 4.4.2 バックアップ・復旧
- **バックアップ方式**: macOS Time Machine によるユーザー側バックアップに委ねる。App Sandbox内データは自動的にバックアップ対象
- **復旧要求**: アプリ再インストール時、Keychainデータは保持される。送付履歴（Core Data）およびアプリ設定（UserDefaults）はアプリ削除で消去される旨をユーザーに通知

#### 4.4.3 監視・ログ
- **監視項目**: スタンドアロンアプリのためサーバーサイド監視なし。App Store ConnectのクラッシュレポートおよびTestFlight経由のフィードバックで品質監視
- **ログ保存期間**: Unified Loggingに依存（macOSの設定に準拠。通常数日〜数週間）

## 5. 制約事項

### 5.1 技術的制約
- libarchiveは7Z書き込みに非対応。7Z形式のサポートは初期スコープから除外する
- TAR系形式（TAR.GZ / TAR.BZ2 / TAR.ZST）はlibarchiveの仕様上、AES-256暗号化に非対応。暗号化はZIP形式のみ
- Gmail API添付ファイル上限は25MB。これを超えるファイルは分割圧縮機能（Phase 4）で対応
- App Sandboxにより、ユーザーが明示的に選択したファイル以外へのアクセスは不可

### 5.2 業務的制約
- メール送信はGmail限定。Outlook / Yahoo Mail等への対応は初期スコープ外
- オフライン環境ではGmail送信機能は使用不可（圧縮・解凍・パスワード生成はオフラインで利用可能）

### 5.3 法的制約
- App Store Review Guidelines への準拠が必要
- Googleのデータ利用ポリシー（Google API Services User Data Policy）への準拠が必要
- プライバシーポリシーの公開が必要（App Store要件 + Google OAuth要件）
- 暗号化ソフトウェアの輸出規制（EAR）への該当確認が必要

### 5.4 予算・スケジュール制約
- Apple Developer Program年間登録料: $99/年
- Google Cloud Platform: Gmail API無料枠内での運用を想定
- 開発期間: 全5フェーズで約5〜8ヶ月（Phase 1〜5）
- App Store審査期間: 2〜4週間を見込む

### 5.5 組織的制約
- 個人開発またはに小規模チーム体制を想定
- macOS開発経験（Swift / SwiftUI）を持つ開発者が最低1名必要

- 有料版・無料版の機能分割ラインは詳細未確定。基本方針として圧縮・解凍は無料、暗号化・Gmail送信は有料
- Phase 1のMVPスコープは圧縮・解凍・AES-256・パスワード生成・基本UIとする（最終確認が必要）

### 7.2 リスク対応策

#### R001: App Storeリジェクト
- **発生条件**: App Store Review Guidelinesへの違反
- **影響範囲**: リリーススケジュールの遅延
- **予防策**: スケジュール送信・7z CLIバンドル・Mail.app連携等のリジェクトリスク項目は設計段階で排除済み。Gmail連携理由の説明文・テスト用Googleアカウント・プライバシーポリシーを事前準備
- **対処策**: リジェクト理由に基づく修正・再申請。Appleとのコミュニケーション
- **代替案**: Mac Developer ID署名による直接配布（App Store外）

#### R002: Google OAuth審査遅延
- **発生条件**: Googleによるアプリ検証が長引く（数週間〜数ヶ月）
- **影響範囲**: Gmail送信機能のリリース遅延
- **予防策**: Phase 3開始前にOAuth consent screen審査を申請。スコープを`gmail.send`のみに限定し審査を簡素化
- **対処策**: Phase 3とPhase 4の順序入れ替え。OAuth審査完了まで他機能を先行開発
- **代替案**: 一時的にテストモード（100ユーザー制限）でリリース
