# SecureZip for Mac - プロジェクト構造と開発計画

## 📋 プロジェクト概要

### 基本情報
- **プロジェクト名**: SecureZip for Mac
- **コードネーム**: SZ-2026
- **バージョン**: 1.0.0
- **作成日**: 2026-02-27
- **更新日**: 2026-02-27
- **ステータス**: 🟢 計画

### 目的とビジョン
**目的**:
コマンドラインの知識がなくても、ビジネスレベルのセキュアなファイル送付ができるmacOSアプリケーションを提供する。GUIのみで暗号化圧縮・パスワード管理・Gmail送信までを一気通貫に完結させ、PPAPワークフローの安全性と利便性を両立する。

**ビジョン**:
非エンジニアのMacビジネスユーザーが、3ステップ以内で暗号化ファイルを安全に送信できる状態を実現する。パスワード自動生成・別送・履歴自動削除により、セキュリティインシデントのリスクを根本から低減する。

**対象ユーザー**:
macOS 13 Ventura以降を使用する非エンジニアのビジネスユーザー。Gmailアカウントを保有し、機密ファイルをメールで送付するケースが多いユーザーを主なターゲットとする。

## 🏗️ システムアーキテクチャ

### 技術スタック
#### アプリケーション
- **プラットフォーム**: macOS 13 Ventura 以降
- **開発言語**: Swift 5.9+
- **UIフレームワーク**: SwiftUI（macOS 13+）
- **アーキテクチャ**: MVVM + Service Layer

#### 圧縮・暗号化
- **圧縮エンジン**: libarchive（macOS標準搭載）
  - 対応形式: ZIP / TAR.GZ / TAR.BZ2 / TAR.ZST
  - 暗号化: AES-256（ZIP形式のみ、`zip:encryption=aes256`オプション）
- **暗号化ライブラリ**: CryptoKit（macOS標準搭載）
  - パスワード生成: SecRandomCopyBytes
  - 公開鍵暗号: Curve25519（Phase 4実装予定）

#### データ永続化
- **ローカルDB**: Core Data（macOS標準搭載）
  - 送付履歴（SendHistory）
  - 送付先（Recipient）
  - アプリ設定（AppSettings）
- **シークレット管理**: Keychain Services（macOS標準搭載）
  - OAuthトークン（Access Token / Refresh Token）
  - 暗号化パスワード（履歴IDと紐付け）

#### 外部連携
- **メール送信**: Gmail API v1（REST API / HTTPS）
- **OAuth認証**: Google Sign-In SDK + GTMAppAuth（SPM経由）
- **認証方式**: OAuth 2.0 + PKCE、スコープ: `gmail.send`のみ

#### 開発環境
- **IDE**: Xcode 15.0+
- **パッケージ管理**: Swift Package Manager（SPM）のみ
- **テストフレームワーク**: XCTest + XCUITest
- **ビルド**: Universal Binary（Apple Silicon / Intel）

## 🔌 外部API仕様

### Gmail API
| メソッド | エンドポイント | 概要 | 認証 |
|---------|--------------|------|------|
| POST | `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` | メール送信 | OAuth 2.0 Bearer Token |
| POST | `https://oauth2.googleapis.com/token` | トークンリフレッシュ | Client ID / Secret |

- **認証スコープ**: `https://www.googleapis.com/auth/gmail.send`（送信のみ、読み取り不可）
- **添付上限**: 25MB（これを超える場合は分割圧縮推奨）
- **リクエスト形式**: Base64URLエンコードされたMIMEメッセージ

### OAuth 2.0認証フロー
1. アプリがOAuth認証画面を開く（Safari経由）
2. ユーザー認証後、カスタムURLスキームでコールバック受信
3. 認証コードをアクセストークン・リフレッシュトークンに交換
4. トークンをKeychain Servicesに保存
5. 以降はGTMAppAuthが自動でトークンリフレッシュ

## ⚠️ リスクと課題

### 技術的リスク
| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| App Storeリジェクト（R001） | 高 | 中 | リジェクトリスク項目の事前排除・App Review Notes準備 |
| Google OAuth審査遅延（R002） | 高 | 中 | Phase 3開始前に早期申請・スコープをgmail.sendのみに限定 |
| libarchive仕様変更（R004） | 高 | 低 | macOSベータ版での事前検証・代替ライブラリ調査 |
| macOS新バージョン非互換（R007） | 中 | 中 | WWDC後に早期検証 |

### 技術的制約事項
- libarchiveは7Z書き込み非対応（7Z形式は対象外）
- TAR系形式はAES-256暗号化非対応（暗号化はZIP形式のみ）
- Gmail API添付ファイル上限: 25MB
- App Sandboxにより、ユーザー選択ファイル以外へのアクセス不可
- Gmail送信のみ対応（Outlook / Yahoo Mail等は対象外）
