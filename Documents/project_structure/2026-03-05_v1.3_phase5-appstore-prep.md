# SecureZip for Mac - プロジェクト構造と開発計画

## プロジェクト概要

### 基本情報
- **プロジェクト名**: SecureZip for Mac
- **コードネーム**: SZ-2026
- **バージョン**: 1.0.0
- **Bundle ID**: com.tkrite.SecureZip
- **作成日**: 2026-02-27
- **更新日**: 2026-03-05
- **ステータス**: 開発中（Phase 5 進行中 -- App Store 申請準備）

### 目的とビジョン
**目的**:
コマンドラインの知識がなくても、ビジネスレベルのセキュアなファイル送付ができるmacOSアプリケーションを提供する。GUIのみで暗号化圧縮・パスワード管理・Gmail送信までを一気通貫に完結させ、PPAPワークフローの安全性と利便性を両立する。

**ビジョン**:
非エンジニアのMacビジネスユーザーが、3ステップ以内で暗号化ファイルを安全に送信できる状態を実現する。パスワード自動生成・別送・履歴自動削除により、セキュリティインシデントのリスクを根本から低減する。

**対象ユーザー**:
macOS 13 Ventura以降を使用する非エンジニアのビジネスユーザー。Gmailアカウントを保有し、機密ファイルをメールで送付するケースが多いユーザーを主なターゲットとする。

## システムアーキテクチャ

### 技術スタック
#### アプリケーション
- **プラットフォーム**: macOS 13 Ventura 以降（現在のビルド設定は macOS 15.0、要引き下げ）
- **開発言語**: Swift 5.9+
- **UIフレームワーク**: SwiftUI（macOS 13+）
- **アーキテクチャ**: MVVM + Service Layer
- **Bundle ID**: com.tkrite.SecureZip
- **多言語対応**: 日本語（ja）・英語（en）

#### 圧縮・暗号化
- **圧縮エンジン**: macOS システム libarchive（`/usr/lib/libarchive.2.dylib`）
  - ZIP: システム libarchive C API経由（AES-256暗号化対応）
  - TAR系: macOS標準CLI（`tar`, `unzip`）をProcess経由で使用（継続）
  - 対応形式: ZIP / TAR.GZ / TAR.BZ2 / TAR.ZST
  - ヘッダー: プロジェクト内蔵（`Infrastructure/Headers/archive.h`, `archive_entry.h`）
- **暗号化ライブラリ**: CryptoKit（macOS標準搭載）
  - パスワード生成: SecRandomCopyBytes

#### データ永続化
- **ローカルDB**: Core Data（macOS標準搭載）
  - 送付履歴（SendHistoryEntity）- recipientEmailを非正規化して直接保存
  - **注意**: `.xcdatamodeld` 不在時はプログラム定義のNSManagedObjectModel + NSInMemoryStoreTypeでフォールバック
- **アプリ設定**: UserDefaults（cancelDelaySeconds, isSeparatePasswordEnabled, postCompressionAction）
- **シークレット管理**: Keychain Services（macOS標準搭載）
  - OAuthトークン: GIDSignIn SDK が自動管理（アプリ側の手動保存は廃止済み）
  - 暗号化パスワード（履歴IDと紐付け）

#### 外部連携
- **メール送信**: Gmail API v1（REST API / HTTPS）
- **OAuth認証**: Google Sign-In SDK（GIDSignIn.sharedInstance.currentUser から直接トークン取得）
- **認証方式**: OAuth 2.0 + PKCE、スコープ: `gmail.send`のみ

#### 開発環境
- **IDE**: Xcode 15.0+（Xcode 16のファイル自動同期機能を活用）
- **パッケージ管理**: Swift Package Manager（SPM）のみ
- **テストフレームワーク**: XCTest + XCUITest
- **ビルド**: Universal Binary（Apple Silicon / Intel）-- Release ビルドで自動生成

## 外部API仕様

### Gmail API（実装済み）
| メソッド | エンドポイント | 概要 | 認証 |
|---------|--------------|------|------|
| POST | `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` | メール送信 | OAuth 2.0 Bearer Token |

- **認証スコープ**: `https://www.googleapis.com/auth/gmail.send`（送信のみ、読み取り不可）
- **添付上限**: 25MB（これを超える場合は分割圧縮推奨）
- **リクエスト形式**: Base64URLエンコードされたMIMEメッセージ
- **トークン管理**: GIDSignIn SDK が自動管理（リフレッシュ含む）。アプリ側での手動トークン保存は廃止済み。

### OAuth 2.0認証フロー（実装済み）
1. アプリがGIDSignIn経由でOAuth認証画面を開く（Safari経由）
2. ユーザー認証後、reversed-client-IDスキームでコールバック受信（AppDelegate処理）
3. GIDSignIn SDKがトークンを自動管理・自動リフレッシュ
4. `GIDSignIn.sharedInstance.currentUser?.accessToken.tokenString` で直接取得
5. SettingsViewModelが起動時にGIDSignIn SDKからconnectedEmailを復元

## リスクと課題

### 技術的リスク
| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| App Storeリジェクト（R001） | 高 | 中 | Privacy Manifest作成・EAR申告・disable-library-validation削除済み |

| Homebrew版ヘッダーとシステムlibarchiveのバージョン差（R005） | 中 | 低 | 主要APIのTBD確認済み。問題発生時はSDK版ヘッダーに切替 |
| macOS新バージョン非互換（R007） | 中 | 中 | WWDC後に早期検証 |
| Privacy Manifest未対応によるリジェクト（R008） | 高 | 高 | Phase 5残タスクとして優先対応が必要 |

### 既知の技術的問題
- **Homebrew版ヘッダーとシステムlibarchiveのバージョン差**: `Infrastructure/Headers/`にはHomebrew版archive.h/archive_entry.hを配置しているが、リンク先はシステム版libarchive。TBDで主要シンボルの存在は確認済みだが、マイナーなAPI不整合のリスクが残る。問題発生時はmacOS SDK版ヘッダー（`/Library/Developer/CommandLineTools/SDKs/MacOSX.sdk/usr/include/archive.h`）への切替を検討する。
- **MACOSX_DEPLOYMENT_TARGET 15.0**: 要件定義書ではmacOS 13以降がサポート対象だが、現在のビルド設定は15.0。システムlibarchiveへの移行により技術的には13への引き下げが可能。Phase 5残タスクとして対応が必要。
- **Privacy Manifest（PrivacyInfo.xcprivacy）未作成**: 2024年5月以降App Store必須。Keychain Services / FileManager等のRequired Reason API使用を申告する必要がある。
- **Core Dataフォールバック**: `.xcdatamodeld`不在時はNSInMemoryStoreTypeで動作
- **UserDefaults vs Core Data設定保存の不統一**: 技術仕様書ではAppSettingsエンティティ（Core Data）で設定保存としているが、現在の実装はUserDefaultsを使用

### 技術的制約事項
- libarchiveは7Z書き込み非対応（7Z形式は対象外）
- TAR系形式はAES-256暗号化非対応（暗号化はZIP形式のみ）
- Gmail API添付ファイル上限: 25MB
- App Sandboxにより、ユーザー選択ファイル以外へのアクセス不可
- Gmail送信のみ対応（Outlook / Yahoo Mail等は対象外）
- キャンセル猶予秒数上限: 10秒（要件定義書の30秒から変更）

### 要件定義書との差異（未解決）
| 項目 | 要件定義書 | 現在の実装 | 対応予定 |
|------|-----------|-----------|---------|
| デプロイターゲット | macOS 13+ | macOS 15+ | Phase 5で引き下げ |
| キャンセル秒数上限 | 1〜30秒 | 1〜10秒 | 仕様書更新 or リリース前に判断 |
| 設定保存先 | Core Data（AppSettings） | UserDefaults | Phase 5前に統一方針を決定 |
