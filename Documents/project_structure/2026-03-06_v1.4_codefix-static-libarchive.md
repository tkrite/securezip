# SecureZip for Mac - プロジェクト構造と開発計画

## プロジェクト概要

### 基本情報
- **プロジェクト名**: SecureZip for Mac
- **コードネーム**: SZ-2026
- **バージョン**: 1.0.0
- **Bundle ID**: com.tkrite.SecureZip
- **作成日**: 2026-02-27
- **更新日**: 2026-03-06
- **ステータス**: 開発中（Phase 5 進行中 -- App Store 申請準備）

### 目的とビジョン
**目的**:
コマンドラインの知識がなくても、ビジネスレベルのセキュアなファイル送付ができるmacOSアプリケーションを提供する。GUIのみで暗号化圧縮・パスワード管理・Gmail送信までを一気通貫に完結させ、PPAPワークフローの安全性と利便性を両立する。

**ビジョン**:
非エンジニアのMacビジネスユーザーが、3ステップ以内で暗号化ファイルを安全に送信できる状態を実現する。パスワード自動生成・別送・履歴自動削除により、セキュリティインシデントのリスクを根本から低減する。

**対象ユーザー**:
macOS 15.0 以降を使用する非エンジニアのビジネスユーザー。Gmailアカウントを保有し、機密ファイルをメールで送付するケースが多いユーザーを主なターゲットとする。

## システムアーキテクチャ

### 技術スタック
#### アプリケーション
- **プラットフォーム**: macOS 15.0 以降（MACOSX_DEPLOYMENT_TARGET=15.0）
- **開発言語**: Swift 5.9+
- **UIフレームワーク**: SwiftUI（macOS 15.0+）
- **アーキテクチャ**: MVVM + Service Layer
- **Bundle ID**: com.tkrite.SecureZip
- **多言語対応**: 日本語（ja）・英語（en）

#### 圧縮・暗号化
- **圧縮エンジン**: Homebrew libarchive 3.x（静的リンク .a）
  - ZIP: Homebrew libarchive C API経由（AES-256暗号化対応）
  - TAR系: macOS標準CLI（`tar`, `unzip`）をProcess経由で使用
  - 対応形式: ZIP / TAR.GZ / TAR.BZ2 / TAR.ZST
  - ヘッダー: プロジェクト内蔵（`Infrastructure/Headers/archive.h`, `archive_entry.h`）
  - 静的リンク依存: liblzma, libzstd, liblz4, libb2（Homebrew .a）+ libbz2, libz, libiconv（システム）
- **暗号化ライブラリ**: CryptoKit（macOS標準搭載）
  - パスワード生成: SecRandomCopyBytes

#### データ永続化
- **ローカルDB**: Core Data（macOS標準搭載）
  - 送付履歴（SendHistoryEntity）- recipientEmailを非正規化して直接保存
  - **注意**: `.xcdatamodeld` 不在時はプログラム定義のNSManagedObjectModel + NSInMemoryStoreTypeでフォールバック
- **アプリ設定**: UserDefaults（cancelDelaySeconds, isSeparatePasswordEnabled, postCompressionAction）
  - キー管理: `SettingsViewModel.UDKey` enum で一元管理
- **シークレット管理**: Keychain Services（macOS標準搭載）
  - OAuthトークン: GIDSignIn SDK が自動管理（アプリ側の手動保存は廃止済み）
  - 暗号化パスワード（履歴IDと紐付け）

#### 外部連携
- **メール送信**: Gmail API v1（REST API / HTTPS）
- **OAuth認証**: Google Sign-In SDK（GIDSignIn.sharedInstance.currentUser から直接トークン取得）
- **認証方式**: OAuth 2.0 + PKCE、スコープ: `gmail.send`のみ

#### 開発環境
- **IDE**: Xcode 15.0+（Xcode 16のファイル自動同期機能を活用）
- **パッケージ管理**: Swift Package Manager（SPM）のみ
- **テストフレームワーク**: XCTest + XCUITest
- **ビルド**: Universal Binary（Apple Silicon / Intel）-- Release ビルドで自動生成

## データベース設計

### エンティティ設計（実装済み）

#### SendHistoryEntity（非正規化設計）
recipientEmailを直接保持し、Recipientエンティティへの参照を持たない設計を採用（JOIN不要でシンプルに）。

| カラム名 | 型 | 制約 | 説明 |
|---------|-----|------|------|
| id | UUID | PK | 一意識別子 |
| recipientEmail | String | NOT NULL | 送付先メールアドレス（非正規化） |
| fileName | String | NOT NULL | 圧縮ファイル名 |
| originalFileNames | String | NOT NULL | 元ファイル名（JSON配列） |
| fileSize | Int64 | NOT NULL, DEFAULT 0 | ファイルサイズ（bytes） |
| format | String | NOT NULL, DEFAULT "zip" | 圧縮形式 |
| isEncrypted | Bool | NOT NULL, DEFAULT false | 暗号化有無 |
| sentAt | Date | NULL | 送信日時 |
| expiresAt | Date | NULL | 自動削除予定日時 |
| status | String | NOT NULL, DEFAULT "created" | ステータス（created/sending/sent/cancelled/failed） |
| createdAt | Date | NOT NULL | 作成日時 |

### CoreDataStack フォールバック
`.xcdatamodeld`が利用できない場合、プログラム定義のNSManagedObjectModel + NSInMemoryStoreTypeを使用するフォールバック機構を実装済み。

### Keychain保存設計
```swift
// OAuthトークン: GIDSignIn SDKが自動管理（アプリ側の手動保存は廃止）
// 暗号化パスワードのKeychain保存は継続
enum KeychainKey: String {
    case passwordPrefix = "com.securezip.password."  // + historyID
}
// kSecAttrAccessible: kSecAttrAccessibleWhenUnlockedThisDeviceOnly
// kSecAttrSynchronizable: false（iCloud Keychainへの同期なし）
```

### UserDefaults保存設計
| キー（UDKey enum） | 型 | デフォルト値 | 用途 |
|-------------------|-----|-------------|------|
| cancelDelaySeconds | Int | 5 | 送信キャンセル猶予秒数（1〜10秒） |
| isSeparatePasswordEnabled | Bool | false | パスワード別送のデフォルト設定 |
| postCompressionAction | String | "keep" | 圧縮後の元ファイル処理（keep/move/delete） |

**注意**: UserDefaults キーは `SettingsViewModel.UDKey` enum で一元管理。AutoDeleteService・SendViewModel すべてこの定数を参照。

| メソッド | エンドポイント | 概要 | 認証 |

| POST | `https://gmail.googleapis.com/gmail/v1/users/me/messages/send` | メール送信 | OAuth 2.0 Bearer Token |

- **認証スコープ**: `https://www.googleapis.com/auth/gmail.send`（送信のみ、読み取り不可）
- **添付上限**: 25MB（これを超える場合は分割圧縮推奨）
- **リクエスト形式**: Base64URLエンコードされたMIMEメッセージ
- **トークン管理**: GIDSignIn SDK が自動管理（リフレッシュ含む）。アプリ側での手動トークン保存は廃止済み。
- **401リトライ**: GIDSignIn.refreshTokensIfNeeded で1回リトライ。resumed フラグで二重 resume 防止済み。

### OAuth 2.0認証フロー（実装済み）
1. アプリがGIDSignIn経由でOAuth認証画面を開く（Safari経由）
2. ユーザー認証後、reversed-client-IDスキームでコールバック受信（AppDelegate処理）
3. GIDSignIn SDKがトークンを自動管理・自動リフレッシュ
4. `GIDSignIn.sharedInstance.currentUser?.accessToken.tokenString` で直接取得
5. SettingsViewModelが起動時にGIDSignIn SDKからconnectedEmailを復元

## リスクと課題

### 技術的リスク
| リスク | 影響度 | 発生確率 | 対策 |
|--------|--------|---------|------|
| App Storeリジェクト（R001） | 高 | 低 | Privacy Manifest作成済み・EAR申告（担当者対応待ち）・disable-library-validation削除済み |
| Google OAuth審査遅延（R002） | 高 | 中 | スコープをgmail.sendのみに限定・プライバシーポリシー公開済み |
| Homebrew版ヘッダーとlibarchiveバージョン差（R005） | 中 | 低 | 静的リンクで同一バイナリに内包。問題時はSDK版ヘッダーに切替 |
| macOS新バージョン非互換（R007） | 中 | 中 | WWDC後に早期検証 |

### 技術的制約事項
- libarchiveは7Z書き込み非対応（7Z形式は対象外）
- TAR系形式はAES-256暗号化非対応（暗号化はZIP形式のみ）
- Gmail API添付ファイル上限: 25MB
- App Sandboxにより、ユーザー選択ファイル以外へのアクセス不可
- Gmail送信のみ対応（Outlook / Yahoo Mail等は対象外）
- キャンセル猶予秒数上限: 10秒
- String.isValidEmailのIDN（国際化ドメイン）非対応

|------|--------|------|--------|

| 設定保存先 | 仕様書Core Data vs 実装UserDefaults | 仕様書をUserDefaultsに更新 | 2026-03-06 |
