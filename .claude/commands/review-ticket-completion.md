# 開発完了レビューコマンド

`./Documents/tickets/` 内の `$ARGUMENTS` で指定されたチケットに基づいて、lead-developer が開発完了のレビューを実施します。

## 実行フロー

### 1. チケット内容の確認

指定されたチケットファイルを読み込み、以下を確認します：

```bash
# チケットファイルの読み込み
cat ./Documents/tickets/$ARGUMENTS
```

確認項目：
- チケットID、タイトル
- 完了条件（受入基準）
- 技術仕様・変更対象ファイル
- 影響範囲

### 2. 実装内容の検証

#### 2.1 変更対象ファイルの確認

チケットに記載された変更対象ファイルが実際に変更されているか確認：

```bash
# Git での変更確認
git diff --name-only HEAD~[コミット数]

# または直接ファイル確認
cat [変更対象ファイル]
```

#### 2.2 完了条件のチェック

チケットの「完了条件（受入基準）」の各項目を順番に検証：

**機能要件チェック**:
- [ ] 各機能要件が実装されているか
- [ ] 仕様通りの動作をするか

**非機能要件チェック**:
- [ ] パフォーマンス基準を満たしているか
- [ ] セキュリティ基準を満たしているか

**テスト基準チェック**:
- [ ] ユニットテストが存在し、パスしているか
- [ ] 統合テストがパスしているか

#### 2.3 コード品質チェック

```bash
# テスト実行（プロジェクトに応じて）
npm test / pytest / etc.

# Lint チェック（プロジェクトに応じて）
npm run lint / flake8 / etc.
```

### 3. レビュー結果の記録

#### 3.1 チケットファイルの更新

チケットの「修正/開発完了レビュー」セクションを更新：

```markdown
### 開発完了レビュー
**レビュー日**: [現在日時]  
**レビュアー**: Claude Code  
**結果**: ✅ 合格 / ❌ 差し戻し

**チェック項目**:
- [x/[ ]] 全完了条件を満たしている
- [x/[ ]] テストが実行され、パスしている
- [x/[ ]] コード品質基準を満たしている
- [x/[ ]] ドキュメントが更新されている

**コメント**:
[レビューの詳細コメント]
```

#### 3.2 進捗履歴の追記

```markdown
| YYYY-MM-DD HH:MM | Review | 開発完了レビュー実施 - [合格/差し戻し] | Claude Code |
```

### 4. レビュー結果の報告

#### 合格の場合

```
✅ 開発完了レビュー: 合格

チケット: [チケットID]
タイトル: [タイトル]

【確認結果】
✓ 機能要件: 全項目実装済み
✓ テスト: 全テストパス
✓ コード品質: 基準を満たしている

【次のステップ】
ユーザーによる最終確認後、チケットをクローズできます。
コマンド: /close-ticket [チケットファイル名]
```

#### 差し戻しの場合

```
❌ 開発完了レビュー: 差し戻し

チケット: [チケットID]
タイトル: [タイトル]

【未完了項目】
- [ ] [未完了の完了条件1]
- [ ] [未完了の完了条件2]

【問題点】
1. [問題の詳細説明]
2. [問題の詳細説明]

【対応が必要な事項】
- [対応内容1]
- [対応内容2]

修正後、再度レビューを実行してください。
```

### 5. ユーザーへの引き継ぎ

レビュー合格後、以下をユーザーに報告：

1. **動作確認依頼**
   - 実際の動作を確認してもらう項目を提示
   - 確認手順を説明

2. **承認依頼**
   - 問題なければチケットクローズの承認を依頼

## 注意事項

- レビューは客観的な基準に基づいて実施する
- 不明点がある場合は「条件付き合格」とせず、ユーザーに確認する
- 差し戻しの場合は、具体的な修正点を明確に示す
- チケットファイルは必ず更新する（履歴として重要）
